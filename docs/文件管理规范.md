# 项目文件管理规范

## 1. 目录结构与文件分类

### 1.1 项目根目录结构

```
f:\project\graduateSoftware\
├── src/                        # 源代码
├── test/                        # 测试代码
├── algorithm/                   # MATLAB 参考算法（只读）
├── config/                      # 配置文件（params.json）
├── data/                        # 测试数据（.gitignore 忽略）
├── logs/                        # 运行时日志（.gitignore 忽略）
├── build/                       # 编译产物（.gitignore 忽略）
├── docs/                        # 项目文档
├── .VS/                         # Visual Studio 临时文件（.gitignore 忽略）
├── twoProjector.ui              # Qt UI 文件
├── twoProjector.sln             # VS 解决方案文件
├── twoProjector.vcxproj         # VS 项目文件
├── twoProjector.vcxproj.props   # VS 属性表（库配置）
├── build.bat                    # 编译脚本
├── .gitignore                   # Git 忽略规则
└── README.md                    # 项目说明
```

## 2. 各目录职责

| 目录 | 用途 | 版本控制 | 清理策略 |
|------|------|--------|--------|
| `src/` | C++ 源代码 | **提交** | — |
| `test/` | 单元与集成测试 | **提交** | — |
| `algorithm/` | MATLAB 参考实现 | **提交** | 只读 |
| `config/` | 默认参数 JSON | **提交** | — |
| `docs/` | 项目文档 | **提交** | — |
| `data/` | 测试数据 | **忽略** | 定期清理（> 100 MB）|
| `logs/` | 运行时日志 | **忽略** | 自动分割（> 10 MB）|
| `build/` | 编译中间/产物 | **忽略** | `clean.bat` 清除 |
| `.vs/` | VS 元数据 | **忽略** | 自动清理 |

## 3. 编译产物目录规范

### 3.1 build/ 目录标准结构

```
build/
├── Debug/
│   ├── obj/                     # 中间文件 (.obj, .pdb)
│   └── twoProjector.exe         # Debug 可执行文件
├── Release/
│   ├── obj/                     # 中间文件
│   └── twoProjector.exe         # Release 可执行文件
└── logs/                        # 编译日志（build_YYYYMMDD_HHMM.log）
```

### 3.2 输出目录配置（在 .vcxproj.props 中定义）

```xml
<!-- Release 配置 -->
<OutDir>$(ProjectDir)build\Release\</OutDir>
<IntDir>$(ProjectDir)build\Release\obj\</IntDir>

<!-- Debug 配置 -->
<OutDir>$(ProjectDir)build\Debug\</OutDir>
<IntDir>$(ProjectDir)build\Debug\obj\</IntDir>
```

**说明**：
- `OutDir`：最终输出文件（.exe, .lib, .dll）
- `IntDir`：中间文件（.obj, .pdb, .ilk）

**禁止出现在项目主目录中的文件**：
```
❌ *.obj    → 应在 build/<Config>/obj/
❌ *.pdb    → 应在 build/<Config>/
❌ *.ilk    → 应在 build/<Config>/
❌ *.exe    → 应在 build/<Config>/
❌ *.tlog   → 应在 build/<Config>/obj/
```

## 4. 日志文件管理

### 4.1 编译日志

- **位置**：`logs/build_YYYYMMDD_HHMM.log`
- **触发**：每次运行 `build.bat` 时自动创建
- **格式**：纯文本，MSBuild 标准输出
- **保留策略**：7 天滚动保留（手动清理）

### 4.2 运行时日志

- **位置**：`logs/app_YYYYMMDD.log`
- **触发**：程序启动时由 `Logger` 单例创建
- **默认配置**（在 `config/params.json`）：
  ```json
  {
    "logging": {
      "level": "INFO",
      "max_file_size_mb": 10,
      "max_files": 5
    }
  }
  ```
- **自动分割**：单文件 > 10 MB 时自动创建新文件

### 4.3 日志格式

```
[2026-03-13 14:30:00.123] [INFO]  ReconstructionPipeline::run
[2026-03-13 14:30:01.456] [WARN]  PhaseDecoder — 低调制度像素数: 1234
[2026-03-13 14:30:02.789] [ERROR] CalibException — 标定文件缺失: D:/calib.raw
```

## 6. build.bat 使用说明

### 6.1 基本命令

```bash
# Release 版本（默认）
build.bat

# Debug 版本
build.bat Debug

# 显式指定 Release
build.bat Release
```

### 6.2 输出

```
[2026-03-13 14:25:00] === Build Started ===
Building Release|x64 ...
...（编译输出重定向至 logs/build_20260313_1425.log）
BUILD SUCCEEDED
[2026-03-13 14:25:15] === Build Succeeded ===
```

### 6.3 自动创建的目录

脚本会自动创建：
- `build/Debug/obj/`
- `build/Release/obj/`
- `logs/`

## 7. 清理脚本（clean.bat）

### 7.1 创建 clean.bat

```batch
@echo off
REM 清理编译产物脚本

echo 清理编译产物...
rmdir /s /q "build" 2>nul
rmdir /s /q ".vs" 2>nul

echo 清理完成
echo 请重新运行 build.bat 重新编译
```

### 7.2 使用方法

```bash
clean.bat   # 清理所有产物
build.bat   # 从零开始编译
```

## 8. 主要文件配置修复清单

| 文件 | 修改内容 | 目的 |
|------|--------|------|

## 9. 快速参考

### 9.1 主目录中不应出现的文件类型

| 类型 | 正确位置 | 规则 |
|------|--------|------|
| `.obj` | `build/<Config>/obj/` | 由 MSBuild 自动生成 |
| `.pdb` | `build/<Config>/` | 调试信息 |
| `.ilk` | `build/<Config>/` | 增量链接 |
| `.exe` | `build/<Config>/` | 可执行程序 |
| `.log` | `logs/` | 手动或自动生成 |

### 9.2 常见问题排查

**Q：编译后 .obj 文件出现在主目录？**
A：检查 `twoProjector.vcxproj.props` 中的 `IntDir` 配置是否缺失或错误

**Q：build.bat 产生的日志在哪里？**
A：`logs/build_YYYYMMDD_HHMM.log`

**Q：如何强制重新编译？**
A：运行 `clean.bat` 后再运行 `build.bat`

## 10. 文件管理最佳实践

✅ **应该做**：
- 定期运行 `clean.bat` 并重新编译（如果遇到奇怪的链接错误）
- 检查 `logs/` 中的编译日志诊断问题
- 监控 `build/` 大小（Debug 配置通常 > 500 MB）

❌ **不应该做**：
- 不要手动删除 `build/` 中的文件（用 `clean.bat`）
- 不要将 `build/` 或 `logs/` 添加到版本控制
- 不要修改 `.obj` 文件（由编译器管理）
- 不要在主目录中放置编译中间文件
